# Product Requirements Document - bmad-cem

**Author:** vkiryakov
**Date:** 2026-03-20

## Executive Summary

CEM-платформа для сбора обратной связи с визуальным граф-конструктором опросов на React Flow. Продукт решает две ключевые проблемы: отсутствие контроля над данными респондентов (хранятся на серверах третьих сторон) и непрозрачная логика бранчинга в существующих инструментах (Typeform, SurveyMonkey, Google Forms). Целевой пользователь — владелец SaaS-продукта, которому необходимо регулярно измерять удовлетворённость клиентов. Данные хранятся на собственном сервере (PostgreSQL), без зависимости от внешних платформ.

### What Makes This Special

Два ключевых отличия: (1) визуальный граф-конструктор, где каждый вопрос — нода с типизированными выходами, а граф определяет порядок и логику прохождения опроса — вместо скрытых правил и списков условий; (2) полный контроль данных на собственном сервере. Дополнительно: встроенная валидация флоу (обнаружение тупиков и циклов), heatmap путей прохождения прямо на графе, типизированные выходы нод (NPS автоматически разветвляется на detractor/passive/promoter).

## Project Classification

- **Тип проекта:** SaaS B2B — веб-приложение (single-user MVP, multi-tenant в перспективе)
- **Домен:** General (CEM/фидбэк) — без высокорегулируемых требований на MVP
- **Сложность:** Low — стандартные требования к безопасности и производительности
- **Контекст:** Greenfield — новый продукт с нуля

## Success Criteria

### User Success

- Создание и управление вопросами через отдельный экран библиотеки вопросов с фильтрацией по типу
- Сборка опроса из готовых вопросов за 2-3 минуты: добавление вопросов на граф через кнопку, настройка бранчинга, запуск
- Completion rate респондентов > 70%
- Респондент проходит опрос без затруднений: mobile-friendly, один вопрос за раз, прогресс-бар

### Business Success

- Полный цикл работает end-to-end: создать вопросы → собрать опрос → добавить респондентов → запустить → собрать ответы → проанализировать
- Валидация на себе за 3 месяца: проведено несколько опросов, получены реальные ответы и инсайты по удовлетворённости клиентов
- Инструмент используется регулярно — не разовый эксперимент

### Technical Success

- Валидация флоу обнаруживает тупики и циклы перед активацией опроса
- Защита целостности данных: вопросы с существующими ответами нельзя редактировать или удалять
- Данные хранятся на собственном сервере (PostgreSQL), без внешних зависимостей

### Measurable Outcomes

- Первый опрос создан и запущен в день деплоя
- Completion rate респондентов > 70%
- Время сборки опроса из готовых вопросов: < 5 минут
- Аналитика отображает NPS, распределение ответов, heatmap путей, satisfaction matrix

## Product Scope & Development Strategy

### MVP Strategy

**Подход:** Problem-solving MVP — минимальный продукт, решающий конкретную проблему: сбор фидбэка с визуальным конструктором и контролем данных.
**Ресурсы:** Solo-developer, full-stack разработка.
**Архитектура:** Single-user, без ролей, без организаций, без подписок, без внешних интеграций.

### MVP (Phase 1)

- **Библиотека вопросов:** CRUD, 5 типов (NPS, открытый, закрытый, матричный, мульти-селект), фильтрация по типу, защита от редактирования при наличии ответов
- **Граф-конструктор опросов:** React Flow, добавление вопросов из библиотеки через кнопку, типизированные выходы, валидация флоу (тупики, циклы)
- **Жизненный цикл опроса:** Draft → Active → Completed → Archived
- **UX респондента:** welcome-экран, пошаговый формат, прогресс-бар, mobile-friendly, навигация только вперёд
- **Распространение:** ручной ввод респондентов (email), персональные ссылки, трекинг статусов и drop-off
- **Аналитика:** NPS дашборд, распределение ответов, heatmap на графе, satisfaction matrix, фильтры
- **Аутентификация:** дефолтный пользователь (admin/123), JWT

### Growth (Phase 2)

- Ролевая модель (Admin, Organization Owner, Organization Member) и multi-tenant
- Мультиязычность (EN/UK/IT)
- Email-рассылка из системы
- Экспорт данных (CSV, PDF)
- Превью опроса перед активацией
- Дублирование опросов

### Vision (Phase 3)

- AI-анализ (sentiment, автокатегоризация)
- Брендирование опросов
- Публичный API, вебхуки
- Дополнительные каналы: in-app виджет, QR, SMS
- Платформа для CEM с расширенной аналитикой и интеграциями

### Risk Mitigation

**Technical:** Граф-конструктор на React Flow — самая сложная часть. Митигация: начать разработку с конструктора, валидировать подход на раннем прототипе.
**Market:** Минимальные — продукт для собственного использования. Валидация: если инструмент используется регулярно через 3 месяца, значит работает.
**Resource:** Solo-developer. Митигация: строгий MVP-скоуп, без расползания фич. Monorepo на Turborepo для эффективной разработки.

## User Journeys

### Journey 1: Владимир — Первый запуск и создание опроса

**Сцена:** Владимир — владелец SaaS-продукта. Фидбэк от клиентов не собирается, потому что нет подходящего инструмента с контролем данных. Он разворачивает CEM-платформу на своём сервере и открывает приложение.

**Путь:**
1. Логин (admin/123) → пустой дашборд с кнопкой «Создать опрос»
2. Переходит в библиотеку вопросов → создаёт набор вопросов: NPS, открытые вопросы про удовлетворённость, закрытые вопросы с вариантами
3. Возвращается на дашборд → нажимает «Создать опрос»
4. В граф-конструкторе добавляет вопросы из библиотеки через кнопку → настраивает бранчинг (NPS detractor → уточняющий вопрос, promoter → благодарность)
5. Запускает валидацию флоу → всё чисто, тупиков нет
6. Добавляет респондентов (email клиентов) → система генерирует персональные ссылки
7. Активирует опрос → вручную рассылает ссылки клиентам

**Кульминация:** Опрос создан за пару минут из готовых вопросов, флоу визуально понятен, ссылки разосланы.

**Результат:** Владимир впервые запустил сбор фидбэка — данные на его сервере.

### Journey 2: Владимир — Анализ результатов

**Сцена:** Прошла неделя, респонденты прошли опрос. Владимир хочет понять, довольны ли клиенты.

**Путь:**
1. Открывает дашборд → видит количество ответов, completion rate
2. Переходит в аналитику опроса → NPS score, распределение detractor/passive/promoter
3. Смотрит распределение ответов по каждому вопросу — гистограммы, pie charts
4. Открывает heatmap на графе → видит, по каким веткам шли респонденты, где был drop-off
5. Изучает satisfaction matrix → находит проблемные зоны продукта
6. Фильтрует по конкретному респонденту → видит его полный путь и ответы

**Кульминация:** Heatmap показывает, что 60% detractors уходят через ветку «поддержка» — инсайт, что поддержка проседает.

**Результат:** Владимир получил конкретные данные для улучшения своего SaaS-продукта.

### Journey 3: Респондент — Прохождение опроса

**Сцена:** Клиент SaaS-продукта Владимира получает персональную ссылку на опрос в мессенджере.

**Путь:**
1. Открывает ссылку на мобильном телефоне → welcome-экран
2. Видит прогресс-бар → первый вопрос (NPS: «Оцените от 0 до 10»)
3. Ставит оценку 6 (detractor) → система направляет по ветке detractor
4. Видит уточняющий открытый вопрос: «Что можно улучшить?» → пишет ответ
5. Далее закрытый вопрос с вариантами → выбирает ответ
6. Финальный экран «Спасибо»

**Кульминация:** Опрос прошёл быстро и без затруднений, бранчинг сработал незаметно — респонденту показали релевантные вопросы.

**Результат:** Ответ сохранён, статус респондента — «завершил».

### Journey 4: Владимир — Ошибка валидации флоу

**Сцена:** Владимир собирает новый опрос со сложным бранчингом и допускает ошибку — одна ветка ведёт в тупик.

**Путь:**
1. Создаёт опрос в граф-конструкторе → добавляет вопросы, настраивает ветвление
2. Нажимает «Активировать» → система запускает валидацию флоу
3. Валидация обнаруживает тупик: нода «Оценка поддержки» при ответе «Нейтрально» не ведёт никуда
4. На графе подсвечивается проблемная нода и ребро
5. Владимир добавляет недостающую связь → повторная валидация → всё чисто
6. Опрос активирован

**Кульминация:** Валидация поймала ошибку до того, как респонденты столкнулись с тупиком.

**Результат:** Опрос запущен без дефектов, все пути проходимы.

## Technical Architecture

- **Frontend:** Next.js + React Flow + shadcn/ui + Tailwind + React Query
- **Backend:** NestJS + TypeORM + PostgreSQL
- **Monorepo:** Turborepo (apps/web, apps/api, packages/ui, packages/shared)
- **API:** REST, internal only (frontend ↔ backend), Swagger/OpenAPI
- **Аутентификация:** Дефолтный пользователь (login: admin, password: 123), JWT-токен, без OAuth/SSO
- **Хранение:** Flow опроса — JSONB в PostgreSQL
- **CI/CD:** GitHub Actions

## Functional Requirements

### Библиотека вопросов

- FR1: Пользователь может создавать вопросы с указанием типа (NPS, открытый, закрытый, матричный, мульти-селект)
- FR2: Пользователь может просматривать список всех вопросов с фильтрацией по типу
- FR3: Пользователь может редактировать вопрос, если на него ещё нет ответов
- FR4: Пользователь может удалить вопрос, если на него ещё нет ответов
- FR5: Система блокирует редактирование и удаление вопросов, на которые уже есть ответы
- FR6: Пользователь может искать вопросы по тексту

### Конструктор опросов

- FR7: Пользователь может создать новый опрос с названием
- FR8: Пользователь может добавлять вопросы из библиотеки на граф через кнопку
- FR9: Пользователь может соединять ноды рёбрами для определения порядка прохождения
- FR10: Пользователь может настраивать бранчинг через типизированные выходы нод (NPS: detractor/passive/promoter, закрытый: выход на каждый вариант)
- FR11: Пользователь может удалять ноды и рёбра с графа
- FR12: Пользователь может настраивать параметры ноды через боковую панель при клике
- FR13: Система визуализирует опрос как направленный граф на React Flow

### Валидация флоу

- FR14: Система обнаруживает тупиковые ноды (ноды без исходящих рёбер, кроме финальных)
- FR15: Система обнаруживает циклы в графе
- FR16: Система подсвечивает проблемные ноды и рёбра на графе
- FR17: Система блокирует активацию опроса при наличии ошибок валидации

### Жизненный цикл опроса

- FR18: Пользователь может активировать опрос из статуса Draft
- FR19: Пользователь может завершить активный опрос (перевод в Completed)
- FR20: Пользователь может архивировать завершённый опрос
- FR21: Система запрещает переактивацию завершённого опроса
- FR22: Система запрещает удаление опросов (только архивация)
- FR23: Пользователь может просматривать список опросов с пагинацией и поиском по названию

### Распространение

- FR24: Пользователь может добавлять респондентов по email через веб-интерфейс
- FR25: Система генерирует персональную уникальную ссылку для каждого респондента
- FR26: Система дедуплицирует респондентов по email в рамках опроса
- FR27: Пользователь может просматривать список респондентов с пагинацией и поиском по email
- FR28: Система отслеживает статус каждого респондента (не открыл / открыл / в процессе / завершил)
- FR29: Система отслеживает drop-off респондентов

### UX респондента

- FR30: Респондент видит welcome-экран при открытии ссылки
- FR31: Респондент проходит опрос в пошаговом формате (один вопрос за раз)
- FR32: Респондент видит прогресс-бар прохождения
- FR33: Система поддерживает навигацию только вперёд (без возврата)
- FR34: Система сохраняет прогресс респондента в localStorage
- FR35: Респондент видит финальный экран «Спасибо» после завершения
- FR36: Интерфейс респондента адаптируется под мобильные устройства
- FR37: Каждый ответ респондента отправляется на backend отдельно

### Аналитика

- FR38: Пользователь может просматривать общий дашборд (количество ответов, completion rate, средний NPS)
- FR39: Пользователь может просматривать распределение ответов по каждому вопросу (гистограмма, pie chart)
- FR40: Пользователь может просматривать satisfaction matrix
- FR41: Пользователь может просматривать heatmap путей прохождения на графе React Flow
- FR42: Пользователь может фильтровать аналитику по дате, респонденту, ветке флоу
- FR43: Пользователь может просматривать полный путь и ответы конкретного респондента

### Аутентификация и UI

- FR44: Система создаёт дефолтного пользователя при первом запуске (admin/123)
- FR45: Пользователь может войти по логину и паролю с получением JWT-токена
- FR46: Система отображает пустой дашборд с кнопкой «Создать опрос» при отсутствии опросов
- FR47: Все списки поддерживают пагинацию
- FR48: Интерфейс использует светлую тему

## Non-Functional Requirements

### Performance

- NFR1: Страницы приложения загружаются за < 2 секунды
- NFR2: Граф-конструктор плавно работает с опросами до 50 нод
- NFR3: Heatmap и аналитика рассчитываются за < 3 секунды для опросов с до 500 ответами
- NFR4: Интерфейс респондента загружает следующий вопрос за < 1 секунду
- NFR5: Валидация флоу выполняется за < 1 секунду

### Security

- NFR6: JWT-токены имеют ограниченное время жизни
- NFR7: API-эндпоинты аутентификации защищены от brute-force (rate limiting на логин)
- NFR8: Ссылки респондентов используют UUID v4 (непредсказуемые)
- NFR9: API респондента не требует аутентификации, но доступен только по персональной ссылке
- NFR10: Данные передаются по HTTPS
